# SLSA Technical Project Policy

**Status:** Draft v0.1 · For Steering Committee review
**Owner:** SLSA Steering Committee (SSC)
**Applies to:** All repositories in the `slsa-framework` GitHub organization

---

## 1. Purpose

The SLSA community has finite maintenance capacity. This policy makes an explicit
commitment about **what the community maintains** and **what it does not**, so that
maintainers are not held responsible for more than the community can realistically
sustain, and so that users have a clear, honest signal about what they can rely on.

The default posture is narrow: the community commits its maintenance capacity to the
**specification** and the **minimum tooling required to make the specification usable**.
Everything else is opt‑in and decided case‑by‑case, or is owned outside the community.

This policy is intended to be simple to apply and simple to enforce. It is used at two
moments: when a new repository is proposed, and during periodic review of existing
repositories (which may result in projects being demoted or archived).

## 2. Project Classification


There are two tiers of projects:

- **Core** — the community commits to maintaining it (see §3).
- **Community** — the community does **not** commit to maintaining it (see §4).

Every repository is classified as either **Core** or **Community** based on the following:

- **Core SLSA project — the community MUST maintain.** Docs & definitions (the specification,
  protobufs/schemas, and documentation) and basic developer tools (libraries, types,
  conformance validators). These are **Core** projects.
- **Discretionary — the SSC decides case‑by‑case.** CLI utilities, and full‑service or
  integration layers (builder workflows, policy verifiers, end‑to‑end ecosystem
  functionality). These may become **Core** only by an explicit SSC decision
  (see §6), otherwise they are **Community** projects.
- **Out of scope.** Anything else (demos, proofs of concept, experiments,
  vendor‑specific integrations, workshop material). These are **Community** projects, which the SLSA community does not maintain.

## 2. Project Requirements

A project must be a Core SLSA project  (or be granted a discretionary exception) **and** meet the
Core requirements to be Core. If either is missing, it is Community project.

## 3. Core tier

A **Core** project is one the SLSA community commits to keep alive. Users may depend on it.

**Requirements (all must hold):**

1. **Maintainers:** at least **two active maintainers from at least two different
   organizations.** This is the load‑bearing requirement and is non‑negotiable.
2. **License:** Apache‑2.0.
3. **Required files:** `README` (stating the tier — see Appendix A), `LICENSE`,
   `MAINTAINERS` (see Appendix B), `SECURITY.md` (with a security contact and a stated
   response commitment), and `CONTRIBUTING`.
4. **Hygiene:** automated tests and CI run on changes; releases are cut on a regular
   cadence; the repository is not stale.
5. **Listed:** the project is recorded in the Core list maintained by the SSC.

If a Core project stops meeting these requirements, it is subject to demotion (§7).

## 4. Community tier

A **Community** project is **not maintained by the SLSA community**. It exists for
reference, demonstration, or experimentation, or it is a former Core project that is no
longer maintained. Users should not assume ongoing support, security response, or
compatibility.

**Requirements (all must hold):**

1. The `README` clearly states that the project is **Community / not maintained by the
   SLSA community** (see Appendix A).
2. If the project is no longer developed at all, it is **archived on GitHub
   (made read‑only).** Read‑only is the default end state for demos, PoCs, workshop
   material, examples, and superseded tools.
3. A Community project may point to an externally maintained successor or fork; if so,
   that successor is listed in `ECOSYSTEM.md` (§5).

Community status is not a judgment of quality. It is a statement about maintenance
commitment only.

## 5. Ecosystem file (`ECOSYSTEM.md`)

The community keeps a **single** `ECOSYSTEM.md` file (a "friends & adopters" list) to
record projects and organizations the community wishes to point to but does **not**
maintain. This keeps the organization lean without losing the ecosystem map. It carries
one `Type` field to distinguish the two things people commonly conflate:

- **Adopter** — an organization or project that *uses* SLSA.
- **Tool** — an externally maintained project the community points to (e.g., a community
  fork of a former Core project, a third‑party builder or verifier).

Entries are added by pull request. The list is illustrative, not an endorsement of
security or fitness for any purpose.

## 6. Decision process

All decisions under this policy are made by a **simple majority (>50%) of Steering
Committee members**, voting on a **GitHub issue** in the `governance` repository.

- The issue is labeled with the relevant action (e.g. `new-project`, `scope-decision`,
  `graduation`, `archival`).
- The issue stays open for **at least 7 days** so members have time to vote.
- The outcome (and tally) is recorded by closing the issue with a summary comment.

> Choice point for the SSC: ">50%" here means a majority of **all** SSC members. If you
> prefer a majority of **votes cast**, change this sentence accordingly.

This single mechanism is used for: admitting a new repository and assigning its tier
(§9), granting a discretionary project Core status (§2), graduating a project
(§8), and demoting/archiving a project (§7).

## 7. Demotion and archival (Core → Community / archived)

A Core project is demoted when it no longer meets the Core requirements (§3) — most
commonly when it drops below two maintainers from two organizations, or goes stale.

**Process:**

1. Any SSC member or maintainer opens an `archival` issue identifying the project and
   the failing requirement(s).
2. The SSC votes (§6).
3. On approval, the project enters a **30‑day grace period** during which the community
   seeks new maintainers or an external owner to adopt it.
4. If no resolution, the `README` is updated to the Community banner (Appendix A), the
   project is archived (read‑only) on GitHub, and — if an external successor exists — it
   is added to `ECOSYSTEM.md`.

## 8. Graduation (Community → Core)

A Community project (or an in‑scope candidate, or a discretionary project the SSC wishes
to commit to) becomes Core through graduation.

**Process:**

1. A maintainer opens a `graduation` issue demonstrating that the project is in scope
   (or requesting a discretionary exception) and meets every Core requirement in §3.
2. The SSC votes (§6).
3. On approval, the project's `README` is updated to the Core banner and it is added to
   the Core list.

## 9. New projects and periodic review

- **Intake:** a new repository is created in the organization only after a `new-project`
  issue is approved by the SSC (§6). The issue assigns the project's scope bucket and
  tier at creation. Out‑of‑scope work should generally live outside the organization and
  be referenced via `ECOSYSTEM.md`.
- **Periodic review:** at least **once per year**, the SSC reviews the entire repository
  list against this policy and opens issues to graduate, demote, or archive projects as
  needed. Any member may also trigger a review of a single project at any time.

## Appendix A — README banners

Place at the very top of each repository's `README`.

**Core:**

```markdown
> **SLSA Core project.** Maintained by the SLSA community. Issues and security reports
> are handled per SECURITY.md. See the SLSA Technical Project Policy for what this means.
```

**Community (still developed, but not a community commitment):**

```markdown
> **SLSA Community project — not maintained by the SLSA community.** Provided as‑is for
> reference/experimentation. No support, security response, or compatibility is implied.
```

**Community (archived / read‑only):**

```markdown
> **Archived & read‑only.** This was a demo / proof of concept / former project and is
> no longer maintained. It is kept for reference only. Do not adopt it for production.
> [If applicable: see ECOSYSTEM.md for a maintained successor.]
```
